# PR 方案评审（codex · 2026-04-09）

## 结论
- 方案方向整体可行，但上线前需补齐认证、限流、存储与告警的落地细节，避免运行期阻断或费用/风控风险。

## 主要风险与建议
1) **发布认证缺口（高）**  
   - 写评论必须 OAuth access_token，当前流程默认用 Cookie 直发会失败。建议：在管控层明确获取/刷新 token 的流程（人工定期更新或内网服务下发），拿不到 token 时一律写入 `pending/` 等待人工，不尝试直发。

2) **Embedding 下载稳定性（中）**  
   - 每次 Action 拉取 BGE 本地模型（~400MB）存在超时风险。建议：增加 HuggingFace 缓存（actions/cache ~/.cache/huggingface）或提供线上 embedding 兜底（如 text-embedding-3-small），按开关选择。

3) **向量库体积膨胀（中）**  
   - 将 `vector_store/`、`reply_index/` 全量提交到 git 会随时间快速膨胀。建议：改存 Actions Cache/Release Asset/对象存储，并设大小上限与定期压缩/重建策略；仓库中仅保留轻量的 hash/metadata。

4) **评论暴增限流（中）**  
   - 仅有 `max_new_comments_per_run`=20，不足以覆盖突发流量。建议：增加每日总量上限、LLM 调用预算上限与风控告警（超阈值直接进入 `pending/`），避免费用或 429。

5) **失败告警路径（中）**  
   - 需明确 401/403/429 或发布失败时的告警渠道（Issue/Slack/邮件），避免仅从日志发现；Cookie/OAuth 失效需快速通知与自检。

## 后续可选增强
- 为回复归档新增周期性清理/压缩任务（如每月重建向量库，移除过期线程），降低存储成本。
- 在 README/plan 中补充运行时开关列表（人工审核、线上 embedding 兜底、限流阈值），方便运维配置。
