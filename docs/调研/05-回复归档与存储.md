# 回复归档与存储调研

> **调研结论摘要**
>
> 存档以 **Markdown + YAML front-matter** 为基础，Git 历史即审计日志。
> 同一用户的追问用**对话线程文件**（thread）管理，保留完整上下文；首次回复和追问在同一文件中追加，避免碎片化。
> 文章内容摘要在首次处理时抓取并缓存，后续作为 LLM 上下文。
> **人工回复权重最高**：真人回复写入 thread 文件后，会同时被索引入向量库作为"高权重示例"，下次遇到相似问题优先引用。
> RAG 从归档中检索历史回复，作为生成新回复的参考样本，形成自学习闭环。

## 1. 设计目标

- **人可读**：Markdown 格式，可在 GitHub 直接浏览
- **机器可查**：结构化 front-matter，便于脚本过滤和统计
- **增量追加**：新回复只新增文件/追加内容，不破坏已有记录
- **版本追踪**：Git commit 历史即审计日志

## 2. 目录结构设计

```
archive/
├── articles/
│   └── {article_id}/
│       ├── meta.md                  # 文章元数据 + 内容摘要
│       └── threads/
│           └── {thread_id}.md       # 一个对话线程（首评 + 所有追问 + 所有回复）
data/
├── seen_ids.json                    # 已处理评论 ID 集合
├── wiki_hash.json                   # CSM Wiki 文件哈希
├── reply_index/                     # 归档回复的向量索引（供 RAG 检索历史）
└── vector_store/                    # CSM Wiki 向量索引
```

### 线程 ID 规则

- 每条**顶级评论**（非回复）生成一个新 thread，`thread_id = comment_id`
- 对该评论的追问（子评论）归入同一 thread
- 顶级回复不一定是时间顺序，需按 `parent_id` 关联

## 3. 对话线程文件格式

一个 thread 包含一次完整的对话（原始评论 + 所有追问/回复，按时间顺序追加）。

文件：`archive/articles/{article_id}/threads/{thread_id}.md`

```markdown
---
thread_id: "12345678"
article_id: "98765432"
article_title: "CSM 最佳实践系列（一）"
article_url: "https://zhuanlan.zhihu.com/p/98765432"
commenter: "用户昵称"
started_at: "2024-04-09T10:30:00+08:00"
last_updated: "2024-04-09T11:15:00+08:00"
turn_count: 3
human_replied: true      # 真人是否参与过本线程
---

## 对话记录

### 2024-04-09 10:30 · 用户昵称（评论 #12345678）

> 请问 CSM 如何处理客户的升级投诉？

### 2024-04-09 10:35 · 机器人回复（model: deepseek-chat, tokens: 342）

针对客户升级投诉，建议遵循以下步骤：首先在 24 小时内确认收到...

*参考来源：csm-wiki/escalation.md § 升级处理流程*

---

### 2024-04-09 11:00 · 用户昵称（追问 #12345999）

> 如果客户坚持要求全额退款怎么办？

### 2024-04-09 11:05 · 真人回复（作者本人）⭐

这种情况需要先评估客户价值和合同条款，通常...

---

### 2024-04-09 11:10 · 用户昵称（追问 #12346001）

> 明白了，感谢！请问有没有相关的文档可以参考？

### 2024-04-09 11:15 · 机器人回复（model: deepseek-chat, tokens: 189）

可以参考我们的 CSM 最佳实践文档...
```

### 关键设计点

- `⭐` 标记真人回复，便于快速辨认
- `human_replied: true` 供脚本快速过滤"有真人参与的线程"
- 每轮对话追加到文件末尾，保留时序
- 同一 thread 的上下文在生成下一条回复时全量传入 LLM（控制在 2000 tokens 以内）

## 4. 追问上下文处理

当检测到新评论是某 thread 的追问时：

```python
def build_conversation_context(thread_file: str) -> list[dict]:
    """将 thread 历史转换为 LLM messages 格式"""
    turns = parse_thread(thread_file)
    messages = []
    for turn in turns[-6:]:  # 最多保留最近 6 轮，控制 token
        role = "assistant" if turn["author"] in ["机器人", "真人"] else "user"
        messages.append({"role": role, "content": turn["content"]})
    return messages

# 生成追问回复时传入历史
response = client.chat.completions.create(
    model=model,
    messages=[
        {"role": "system", "content": system_prompt},
        *build_conversation_context(thread_file),   # 历史轮次
        {"role": "user", "content": new_comment},   # 最新追问
    ]
)
```

> 历史截取最近 6 轮（约 1000~1500 tokens），超过部分可先摘要再注入。

## 5. 文章内容作为上下文

首次处理某文章时，抓取文章正文并生成摘要，存入 `meta.md`：

```markdown
---
article_id: "98765432"
title: "CSM 最佳实践系列（一）"
url: "https://zhuanlan.zhihu.com/p/98765432"
summary: "本文介绍了 CSM 在客户成功管理中的核心方法论，包括客户健康评分、QBR 流程..."
fetched_at: "2024-04-01T00:00:00Z"
monitored_since: "2024-04-01"
total_threads: 12
---
```

- 摘要由 LLM 自动生成（首次抓取时一次性调用，之后复用）
- 回复评论时，将 `summary` 注入 System Prompt，帮助 LLM 理解文章背景
- 摘要控制在 200 tokens 以内，几乎不增加费用

## 6. 人工回复的权重机制

```
真人回复 → 写入 thread.md（标记 ⭐）→ 索引到 reply_index 向量库
                                         ↓
下次遇到相似问题 → RAG 检索 reply_index → 找到真人回复示例
                                         ↓
                               注入 Prompt 作为高权重参考：
                               "以下是作者本人的回复示例（最高参考权重）：..."
```

### 真人回复索引规则

```python
def index_human_reply(thread: dict, reply_text: str):
    """将真人回复以高权重写入向量库"""
    doc = {
        "text": f"[问题] {thread['question']}\n[作者回复] {reply_text}",
        "source": "human_reply",
        "weight": "high",
        "article_id": thread["article_id"],
        "thread_id": thread["thread_id"],
    }
    reply_index.add_documents([doc])
```

### 检索时的权重排序

```python
def retrieve_with_human_priority(query: str, k: int = 4) -> list[str]:
    """先检索真人回复，再补充 Wiki 内容"""
    human_results = reply_index.similarity_search(
        query, k=2, filter={"weight": "high"}
    )
    wiki_results = wiki_vectorstore.similarity_search(query, k=2)
    return human_results + wiki_results  # 真人回复排前面
```

> 通过排序和 Prompt 中的明确说明，保证真人回复风格和内容优先被 LLM 学习和模仿。

## 7. Git 提交策略

```bash
# 每次 Action 运行结束后提交
git add archive/ data/seen_ids.json data/reply_index/
git diff --cached --quiet || \
  git commit -m "bot: reply to $(git diff --cached --name-only | wc -l) new comments [skip ci]"
git push
```

- `[skip ci]` 避免存档提交再次触发 workflow
- `data/reply_index/` 随归档一起提交，保持同步

## 8. 状态文件：seen_ids.json

```json
{
  "articles": {
    "98765432": {
      "processed_comment_ids": ["12345678", "12345999"],
      "threads": {
        "12345678": "archive/articles/98765432/threads/12345678.md"
      }
    }
  },
  "last_run": "2024-04-09T11:15:00Z"
}
```

- 同时记录 comment_id → thread 文件的映射
- 支持快速判断新评论属于已有 thread 还是新 thread
