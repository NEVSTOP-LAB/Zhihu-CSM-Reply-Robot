# 知乎数据获取调研

> **调研结论摘要**
> 
> 知乎无官方开放 API，所有方案均基于逆向 Web API v4。
> 推荐方案：**直接调用 API v4 + Cookie 认证**（最稳定、最适合 GitHub Actions 无头环境）。
> CLI 工具（zhihu-cli / opencli）适合本地交互使用，但在无头服务器上均依赖 Cookie，与直接调用 API 本质相同。
> 写操作（发布评论）需要 OAuth access_token，Cookie 不可用，这是最大的技术挑战。
> Cookie 失效是最主要的运维风险，建议定期手动更新并通过告警机制提醒。

## 1. 官方 API 现状

知乎**没有公开的官方 API**，但其网页端使用 RESTful v4 接口，可通过以下两种方式访问：

| 方式 | 认证 | 适用场景 | 风险 |
|------|------|----------|------|
| Cookie（z_c0 + \_xsrf） | 浏览器登录后抓包导出 | 个人/小规模 | 账号封禁风险 |
| OAuth 2.0 | 开放平台注册应用 | 正式/高频 | 需审核，有限配额 |

> 推荐路径：使用 Cookie 认证做 PoC，后期如有必要再申请 OAuth。

## 2. 核心读取接口（API v4）

```
# 专栏文章评论（分页，最大 limit=20）
GET https://www.zhihu.com/api/v4/articles/{article_id}/comments
    ?limit=20&offset=0

# 问题下回答列表
GET https://www.zhihu.com/api/v4/questions/{question_id}/answers

# 回答下评论
GET https://www.zhihu.com/api/v4/answers/{answer_id}/comments

# 用户信息
GET https://www.zhihu.com/api/v4/members/{url_token}
```

返回 JSON 包含 `data[]` 和 `paging.is_end`，需循环翻页直到 `is_end=true`。

参考：[知乎 API v4 整理](https://yifei.me/note/460) | [博客园整理版](https://www.cnblogs.com/apachecn/p/17313135.html)

## 3. 写操作接口

```
# 发表评论（一级 / 楼中楼回复加 parent_id）
POST https://api.zhihu.com/v4/comments
Body: { content, object_id, object_type: "article"|"answer", [parent_id] }
Header: Authorization: Bearer {access_token}

# 发布回答
POST https://api.zhihu.com/v4/answers
Body: { question_id, content }
```

> 写操作**必须**使用 OAuth access_token，Cookie 方式无法调用写接口。

## 4. CLI 工具对比与推荐

### 4.1 三种方案对比

| 维度 | BAIGUANGMEI/zhihu-cli | opencli（OpenClaw Skill） | 直接调用 API v4 |
|------|-----------------------|--------------------------|-----------------|
| **安全性** | 中等：Cookie 存本地，不经第三方 | 高：复用本地 Chrome 会话，不暴露密码 | 高：Cookie 自管，全掌控 |
| **封号风险** | 中：有降低风控设计，但非零 | 中低：行为最接近真实浏览器 | 中：取决于请求频率和 UA 设置 |
| **掉线频率** | 中：Cookie 手动更新，知乎改版可能失效 | 低：浏览器会话自动同步，失效即重新登录 | 中：Cookie 有效期 1~6 个月，到期需人工更新 |
| **无头/CI 环境** | ✅ 支持（Cookie 方式） | ❌ 需要本地 Chromium 桌面环境 | ✅ 完全支持 |
| **GitHub Actions** | ✅ 可用 | ❌ 不适合（需本地浏览器） | ✅ 最佳选择 |
| **写操作（发评论）** | ❌ 受限，需 OAuth | ❌ 受限 | ❌ 同样需要 OAuth |
| **维护状态** | 活跃（2025年有 commit） | 活跃（企业级平台） | N/A（自维护） |
| **学习成本** | 低 | 高（OpenClaw 生态） | 中（需理解 API 结构） |

### 4.2 各工具详情

**BAIGUANGMEI/zhihu-cli**（[GitHub](https://github.com/BAIGUANGMEI/zhihu-cli) | [PyPI](https://pypi.org/project/zhihu-cli/)）
- 专为知乎设计，`pip install pyzhihu-cli` 即用
- 统一 User-Agent、CSRF token 处理，主动降低风控风险
- Cookie 登录：`zhihu login --cookie "z_c0=xxx; _xsrf=yyy"`
- 适合知乎专项脚本化读取，不依赖桌面环境

**opencli / OpenClaw Skill**（[llmbase.ai](https://llmbase.ai/openclaw/zhihu-cli/) | [zeeklog.com](https://zeeklog.com/opencli-ba-ren-he-wang-zhan-bian-cheng-ming-ling-xing-gong-ju-6500-star-de-ai-yuan-sheng-cli-shu-niu/)）
- 复用本地 Chrome 登录态，零 Cookie 管理负担
- 封号风险最低（行为与真实浏览器完全一致）
- 致命缺点：**需要本地 Chromium 桌面环境**，无法在 GitHub Actions 无头容器中运行
- 适合本地测试和调试，不适合自动化 CI/CD

**直接调用 API v4**
- 无额外依赖，代码完全自控
- Cookie 存于 GitHub Secrets，安全性最高
- 结合 `requests.Session` 持久化 Cookie，加 User-Agent 伪装
- **推荐用于 GitHub Actions 自动化流程**

### 4.3 推荐方案

```
GitHub Actions 生产环境：直接调用 API v4 + Cookie（存 Secrets）
本地开发/调试：可用 zhihu-cli 辅助验证接口返回
```

> **核心推荐：直接调用 API v4**。所有 CLI 工具在无头 CI 环境下的本质都是带 Cookie 的 HTTP 请求，自己封装反而更可控、更轻量。opencli 虽然封号风险最低，但无法在 GitHub Actions 中运行，不适合本项目。

## 5. 反爬与长期稳定访问

知乎反爬机制严格，核心策略：

| 问题 | 对策 |
|------|------|
| Cookie 过期 | 加密存储 Cookie 于 GitHub Secrets；设置检测逻辑，失效时告警（自动续签难度大） |
| 频率限制（429） | 请求间随机延迟 1~3s；指数退避重试；本项目为低频轮询，风险低 |
| IP 封禁 | GitHub Actions IP 每次不同，天然规避；如被封可切换 runner |
| 行为识别 | 带 User-Agent（浏览器 UA）、Referer；不并发抢，顺序读取 |

**关键：** Cookie 中 `z_c0` 是身份凭证，需作为 GitHub Secret 存储，**绝不提交到代码**。

## 6. 数据获取流程

```python
# 伪代码：增量评论获取
seen_ids = load_seen_ids("data/seen_comments.json")
for article_id in config["articles"]:
    comments = fetch_all_comments(article_id, cookie)
    new = [c for c in comments if c["id"] not in seen_ids]
    if new:
        process_and_reply(new)
        seen_ids.update(c["id"] for c in new)
save_seen_ids(seen_ids)
```

## 7. 速率限制参考

| 认证类型 | 大致限制 |
|----------|----------|
| Cookie（普通用户） | ~60–100 次/分钟 |
| OAuth 应用 | ~500–1000 次/分钟 |

本项目每次 Action 仅需读取少量文章评论，远低于限制，无需代理池。
