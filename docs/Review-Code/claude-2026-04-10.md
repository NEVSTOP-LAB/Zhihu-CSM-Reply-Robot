### 2.1 `scripts/zhihu_client.py`（AI-003）

**✅ 实现正确的部分**
- `_extract_xsrf` 用正则从 Cookie 字符串提取 `_xsrf`，逻辑正确，覆盖了缺失情况（返回 `None` → `post_comment` 提前返回 `False`）。
- `_request_with_retry` 对 401/403 不重试直接抛 `ZhihuAuthError`，对 429 做指数退避（`2**attempt + random`），行为符合计划要求。

- `_parse_comment` 对 `reply_comment` 字段做了防御性 `.get()`。

**⚠️ 具体问题**

**问题 1：`_request_with_retry` 末尾无论何种原因耗尽重试，都抛 `ZhihuRateLimitError`（第 228 行）**

```python
# 当前
raise ZhihuRateLimitError(
    f"请求失败，已重试 {self.max_retries} 次: {last_exception}"
)
```

当循环因为 `requests.RequestException`（如 DNS 失败、超时）耗尽重试时，仍然抛出 `ZhihuRateLimitError`，但上层调用者（`run_bot.py:706`）会将 `ZhihuRateLimitError` 解读为"知乎限流"并发出限流告警，误导排查。建议在此处改抛更通用的异常（如 `ZhihuRequestError`），或在末尾判断 `last_exception` 类型再选择要抛的异常。

**问题 2：`post_comment` 吞噬了 `ZhihuAuthError`（第 392-396 行）**

```python
except ZhihuAuthError:
    logger.error("评论发布失败: Cookie 失效或权限不足")
    return False
```

发布时遇到 401/403 返回 `False` 而不抛异常，这本身是计划中的设计（让主流程写入 `pending/`）。但实际情况是主流程在 `assess_risk == "safe"` 时调用 `post_comment`，发布失败后只会写入 `pending/`，**不会**触发 `ZhihuAuthError` 的告警路径（`run_bot.py:701-705`），导致 Cookie 失效在发布路径上无声无息。这与流程图中 "Cookie 失效 → 创建 GitHub Issue 告警" 的设计不符。

### 2.2 `scripts/rag_retriever.py`（AI-005）

**✅ 实现正确的部分**
- MD5 增量更新逻辑正确：旧向量先删再插入，已删除文件会清除对应向量。
- `index_human_reply` 使用 `upsert`（幂等），避免重复索引。
- 延迟加载（Lazy Loading）本地模型和线上客户端，避免不必要的依赖加载。

**⚠️ 具体问题**

**问题 3：余弦相似度计算假设向量已 L2 归一化，但未在集合元数据中强制指定距离函数（第 379、403 行）**

```python
# L2 距离 → 余弦相似度（仅对归一化向量成立）
similarity = 1 - (dist ** 2) / 2
```

此公式仅在 ChromaDB 使用 L2 距离且输入向量已归一化时成立。BGE 本地模型使用了 `normalize_embeddings=True`，公式成立；但线上 OpenAI embedding 未经归一化处理，此时公式计算出的"相似度"会偏离真实余弦相似度，导致阈值过滤（`threshold=0.72`）失效。建议：
- 在 `get_or_create_collection` 时显式设置 `metadata={"hnsw:space": "cosine"}`，并对线上 embedding 结果归一化后再存储；或
- 在在线 embedding 模式下，直接用 ChromaDB `cosine` 距离（`1 - dist`）而非当前公式。

**问题 4：`sync_wiki` 在 embedding 失败时会静默标记文件为"已处理"（第 313 行）**

```python
self._wiki_collection.add(...)
updated_count += 1
```

外部 `for` 循环未捕获 `embedding_fn.embed(texts)` 的异常，若某个文件的 embedding 失败（如线上 API 超限），异常会冒泡并中断整个 `sync_wiki`，而此时 `_save_wiki_hashes` **不会被调用**（第 328 行在循环外），所以此文件下次仍会重试——这是正确的行为。但如果在 `add()` 之前失败，旧向量已被删除（第 284-290 行），此文件在本轮运行中处于"旧向量已删，新向量未入"的状态，直到下次运行才会修复。可以考虑先写入再删除（upsert），但当前 ChromaDB API 仅支持 `add/delete` 分步。

**问题 15：`run_articles()` 中 `ZhihuAuthError` 的状态码固定传 401（第 704 行）**

## 三、GitHub Actions Workflows

### 3.1 `bot.yml`

**⚠️ 具体问题**

**问题 18：调度频率为每 15 分钟（`*/15 * * * *`），与计划文档中的每 6 小时不符**

`plan/README.md` 流程图和 AI-004 验收条件均写明 `cron '0 2,8,14,20 * * *'`（每 6 小时），但实现为 `*/15 * * * *`（每 15 分钟）。每 15 分钟触发的后果：
- GitHub Actions 免费额度（2000 分钟/月）可能被大量消耗，即使每次运行只需 2 分钟，每天也消耗 192 分钟，一个月接近 6000 分钟。
- 每次运行都会进行一次知乎 API 请求，15 分钟间隔非常激进，可能加速 Cookie 失效或触发频率限制。
- `bot.yml` 注释中也有矛盾：注释写"每15分钟运行一次"，而计划文档写每6小时。

**问题 19：向量库缓存 key 使用 `${{ github.run_number }}`（第 66-67 行）**

```yaml
key: vectors-${{ runner.os }}-${{ github.run_number }}
restore-keys: |
  vectors-${{ runner.os }}-
```

`github.run_number` 每次运行递增，导致每次 cache 都是新 key，永远 cache miss，然后通过 `restore-keys` 匹配上一次的缓存恢复。这实际上每次都在读取上次缓存（恢复后修改），写入新 key，但旧 key 不会过期，会持续积累缓存条目。计划文档（AI-011）建议用日期作为 key（`${{ hashFiles('...') }}` 或 `${{ env.date }}`），每日更新一次即可，而非每次 run 都创建新条目。
